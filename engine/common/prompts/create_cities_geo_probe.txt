Определи, какие формальные географические ограничения можно предложить пользователю для фильтрации городов SQL-условием к базе городов Германии.

Вход:
- `geo`
- `product`
- `company`
- `available_geo_fields`

Поиск возможен и разумен только по:
- земле
- населению
- месту через `lat` и `lon`

Что есть в базе:
- `state_name`
- `name`
- `pop_total`
- `lat`
- `lon`

Построение SQL для поиска по федеральной земле (`federal state`, `Bundesland`):
- используй для земли только канонические значения `state_name`:
  - `Baden-Württemberg`
  - `Bayern`
  - `Berlin`
  - `Brandenburg`
  - `Bremen`
  - `Hamburg`
  - `Hessen`
  - `Mecklenburg-Vorpommern`
  - `Niedersachsen`
  - `Nordrhein-Westfalen`
  - `Rheinland-Pfalz`
  - `Saarland`
  - `Sachsen`
  - `Sachsen-Anhalt`
  - `Schleswig-Holstein`
  - `Thüringen`
- для земли строй SQL по `state_name`
- используй `ILIKE`
- для одной земли допустим вид:
  - `state_name ILIKE '<state>'`
- для нескольких земель допустим вид:
  - `state_name ILIKE ANY(ARRAY['<state_1>','<state_2>'])`

Построение SQL для поиска по городу:
- для города строй SQL по полю `name`
- для одного города используй такой вид условия:
  - `(name ILIKE '<city>' OR name ILIKE '<city>,%')`
- для нескольких городов используй такой вид условия:
  - `name ILIKE ANY (ARRAY['<city_1>','<city_1>,%','<city_2>','<city_2>,%'])`
- если в названии есть umlaut / ß, нормализуй к немецкой форме, которая должна стоять в SQL:
  - `ae` -> `ä`
  - `oe` -> `ö`
  - `ue` -> `ü`
  - `ss` может соответствовать `ß`, если это нормальная немецкая форма названия

Построение SQL для поиска по населению:
- для `малые города` используй:
  - `pop_total >= 12000 AND pop_total <= 49999`
- для `средние города` используй:
  - `pop_total >= 50000 AND pop_total <= 199999`
- для `крупные города` используй:
  - `pop_total >= 200000 AND pop_total <= 499999`
- для `крупнейшие города` используй:
  - `pop_total >= 500000 AND pop_total <= 4000000`

Построение SQL для поиска по расстоянию / радиусу:
- для расстояния и радиуса строй SQL по `lat` и `lon`
- используй расстояние в километрах
- не определяй и не подставляй координаты центра самостоятельно
- если центр задан городом, в `sql_condition` используй только такой технический контракт:
  - `__RADIUS_FROM_CITY__('<city>', <radius_km>)`
- `<city>` должен быть нормализован по правилу построения поиска по городу
- `<radius_km>` должен быть целым числом в километрах
- если центров несколько, объединяй такие условия через `OR`
- не используй для радиуса никакие другие формы `sql_condition`

Что считать формальным ограничением:
- `только`
- `кроме`
- `исключить`
- `не включать`
- `в радиусе`
- `не дальше`
- `в пределах`
- любые другие формулировки явного отсечения

Что не считать формальным ограничением:
- высокий / средний / низкий приоритет
- предпочтительно
- желательно
- просто перечисления земель, городов и регионов без явного отсечения
- все, что нельзя надежно выразить через землю, население или координаты

Задача:
- прочитать `geo`, `product`, `company`
- предложить только такие ограничения, которые можно выразить SQL-условием
- если ограничение однозначно есть и его можно предложить отдельно, вернуть его в `yes_no_options`
- если ограничение однозначно есть, но есть несколько взаимоисключающих трактовок, вернуть их в `radio_questions`
- если уверенность меньше `85%`, ничего не предлагать

Правила для `yes_no_options`:
- если уверенность `95%` и выше, вернуть checkbox-предложение с `checked=true`
- если уверенность `85%` и выше, но ниже `95%`, вернуть checkbox-предложение с `checked=false`
- если уверенность ниже `85%`, не возвращать такое предложение вообще
- у каждого предложения сразу должен быть `sql_condition`

Правила для `radio_questions`:
- использовать только если ограничение есть, но неясно, какая трактовка правильная
- каждый логически независимый вопрос возвращай отдельным элементом массива
- если наиболее вероятная трактовка уверена на `95%` и выше, отметить ее `checked=true`
- если ограничение есть, но уверенность только `85%` и выше, а не `95%`, вариант `Не применять это ограничение` должен быть `checked=true`
- всегда добавить вариант `Не применять это ограничение`
- у всех обычных вариантов сразу должен быть `sql_condition`
- у варианта `Не применять это ограничение` `sql_condition` должен быть пустой строкой `""`

SQL-правила:
- возвращай только условие для секции `WHERE`, без слова `WHERE`
- используй только поля из доступных и только то, что можно выразить по земле, населению и координатам
- не используй поля вне этого списка

Формат ответа:
- верни только JSON
- структура ответа:
  `{"yes_no_options":[],"radio_questions":[]}`

Структура `yes_no_options`:
- `label`
- `checked`
- `sql_condition`

Структура `radio_questions`:
- `options`

Структура `radio_questions[].options`:
- `label`
- `checked`
- `sql_condition`

Требования:
- не возвращай markdown
- не возвращай пояснения
- не возвращай комментарии
- не возвращай поля вне указанной структуры
- `label` должен быть человекочитаемым текстом для интерфейса, а не служебным SQL-текстом, не используй в `label` служебные фрагменты вроде `state_name`, `name`, `pop_total`,`lat`, `lon`, `ILIKE`, `ARRAY`, `AND`, `OR`
- `label` должен содержать слова вроде "только", "исключительно", чтобы подчеркнуть смысл исключения всего остального.
- если предложение или radio-вариант построен на стандартной шкале размеров городов, в `label` обязательно указывай диапазон населения для каждого типа в скобках.
